# Документация: как поддерживать

## Правило
Изменил API или поведение backend -> в этом же PR обнови оба файла:
- `docs/API.md` (EN)
- `docs/listly-backend-api-ru.md` (RU)

## Правило локализаций
- Оба файла должны описывать один и тот же контракт API.
- Допустимые различия между EN и RU версиями: только язык текста.
- Структура разделов, endpoint'ы, примеры, статусы и ошибки должны быть эквивалентны.

## Минимальный процесс
1. Перед задачей: проверь, затрагивает ли она контракт API.
2. Если да: обнови `docs/API.md` и `docs/listly-backend-api-ru.md` (request/response/errors/enums).
3. Запусти `./gradlew test`.
4. В описании PR добавь секцию `API changes`:
- что изменилось
- обратная совместимость (да/нет)
- что нужно поменять в Android-клиенте

## Версионирование API
- Текущая версия контракта: `v1`.
- Неломающие изменения:
- добавляй новые endpoint'ы/поля без удаления старого поведения.
- Ломающие изменения:
1. Подготовь `v2` контракт (EN+RU).
2. Опиши план миграции клиентов.
3. После миграции удаляй `v1` только отдельным изменением.

## Definition of Done для backend-задачи
- Код реализован
- Тесты зелёные
- `docs/API.md` и `docs/listly-backend-api-ru.md` актуальны
- Логи не содержат секретов
